# 批量测试与 VPS 运行指南

## 1. 当前状态结论

- 当前项目已支持将账号信息写入 `key/accounts.json`（运行时自动创建 `key` 目录）。
- 本地单次注册流程已验证可跑通。
- 批量测试可以开始执行，建议先从小批次做稳定性验证，再扩大并发。

## 2. 如何进行批量测试

### 2.1 推荐先做小批次冒烟

先用 2~3 个任务验证邮箱服务、验证码解析、AWS 注册链路是否稳定。

示例（在项目根目录）：

```bash
python -m src.runners.batch_run
```

如果要自定义并发数量（例如 3），可用：

```bash
python -c "import sys; from pathlib import Path; sys.path.insert(0, str(Path('src'))); from runners.batch_run import batch_run; batch_run(3)"
```

### 2.2 批量测试建议

- 先串行或低并发（1~3）验证成功率。
- 记录每批次成功数、失败原因、平均耗时。
- 临时邮箱服务受限时，加入重试和退避。
- 防止站点风控：避免过高并发、固定节奏请求。

## 3. VPS 是否可行

结论：可行，但需要满足“可用浏览器环境 + 显示能力（或无头模式兼容）+ 稳定网络”。

### 3.1 为什么 VPS 可能遇到问题

本项目使用 Selenium 驱动浏览器自动化，默认会启动真实浏览器窗口。本地可直接显示，VPS 常见是无图形界面环境。

### 3.2 可行方案

- 方案 A：有桌面环境的 VPS（带 GUI），直接运行浏览器自动化。
- 方案 B：无 GUI 的 Linux VPS，使用虚拟显示（如 Xvfb）提供显示环境。
- 方案 C：切换为 headless 模式（需验证目标站点在 headless 下是否被风控或行为变化）。

### 3.3 VPS 落地清单

- 安装 Python 与依赖（`pip install -r requirements.txt`）。
- 安装浏览器与对应 WebDriver（Chrome/Edge 版本需匹配）。
- 准备可写目录与日志目录。
- 配置时区、语言、字体（影响页面渲染与自动化稳定性）。
- 若无 GUI：安装并启用 Xvfb。

## 4. 推荐发布流程

1. 本地先连续跑 3~5 批小规模任务。
2. 确认成功率后，将并发逐步调高。
3. 上 VPS 先跑单次，再跑小批，再放大。
4. 保留失败截图和日志，按错误类型分组修复。

## 5. 风险与注意事项

- 临时邮箱站点接口可能变更，需保持兼容处理。
- 目标注册站点有风控策略，并发过高会降低成功率。
- `key/accounts.json` 含敏感信息，不建议入库。
